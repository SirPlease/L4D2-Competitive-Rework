插件流程

入口文件负责胶水，拆分模块负责算法。运行时是两条常驻线：`OnGameFrame` 处理刷出队列，`Timer_TeleportTick` 每秒处理传送监督。

OnPluginStart创建 CVar、队列、状态，加载 gamedata patch。
Nav 初始化`BuildNavIdIndexMap`、`BuildNavBuckets`，优先读 KV 缓存。
Hook 事件round_start、player_spawn、player_death、ability_use 等。
离开安全区外部 `text.sp` 调 `sm_startspawn`，启动第一波。
双循环`OnGameFrame` 刷特；`Timer_TeleportTick` 传送。
Stop/MapEnd清队列、timer、Nav 运行缓存、死亡 CD。
审查提示：`ResetMatchState()` 当前没有清 `siAlive[]`，报告中标为 P2；成功刷出后重复计数标为 P1。

刷特流程

普通刷特主链路

StartWave重算 cap，收集存活生还，`siQueueCount += limit`，ring 从 `SpawnMin` 开始。
MaintainSpawnQueueOnce按 alive/cap 缺口补队列，稀缺度优先，检查死亡 CD 和支援门槛。
TryNormalSpawnOnce看队首类别，过限类、死亡 CD、支援锁后开始找点。
FindSpawnPosViaNavArea从目标生还者进度桶附近扫描候选 NavArea。
DoSpawnAt成功则记录刷点、降 ring、出队；失败则扩圈。
fallback@max扩到上限后走 `L4D_GetRandomPZSpawnPosition` 兜底。

下一波判定

Timer_CheckSpawnWindow 每秒`lastSpawnSecs++`，同时刷新 anti-baiter。
开始判定时间`DifficultyStrategy_CanEvaluateWave()`，只允许观察补波条件。
普通补波 floor`DifficultyStrategy_CanStartNormalWave()`，是开新波的硬下限。
补波条件全杀手死亡、低存活压力、latest、Pressure after force。
0.1s timer只做异步开波，不代表 0.1 秒刷新。
已核对：`AntiBait_ShouldStartWaveByLatest()` 和 `AntiBait_ShouldStartWaveEarly()` 内部都要求普通补波 floor，Pressure 默认不能绕过 floor。

特感找位流程

找位入口是 `FindSpawnPosViaNavArea(zc, targetSurvivor, ring, teleportMode, ...)`。桶模式和全图模式最终都走同一个候选点评估函数。

生还数据`GetSurPosData` 收集存活且未倒地生还者眼位和 flow。
中心进度桶目标有效取目标 flow，否则取最高 flow 生还者。
桶窗口ring 越大，窗口从 ±6 动态扩到 ±12。
候选过滤冷却、flags、距离环、分散、身前身后、卡壳、可视、路径、桶占比。
评分距离、高度、Flow、分散度加权求总分。
选点TopK 取最高分；FirstFit 开启时达到阈值即返回。
Flow 分桶
预建 0..100 桶，badflow 可映射到附近有效桶。候选点允许 mapped badflow，但会扣 Flow 分。
扩圈策略
普通刷特从 250 起，每失败一次加 100。Boomer/Spitter 支援类上限额外收在 1200。
路径检查
`PathPenalty_NoBuild` 用 NavAreaBuildPath，limitCost 是 `min(ring*3, spawnmax*1.5)`。

特感权重设计流程

total = w_dist[class] * scoreDist + w_hght[class] * scoreHeight + w_flow[class] * scoreFlow + w_disp[class] * scoreDisp
scoreDist = 以各类特感甜点距离为峰值；低难度只放宽 width，默认不移动 sweet
scoreFlow = 候选桶 - 目标桶；前方 1..5 桶最优，太靠后或太靠前都会降分
特感 w_dist w_hght w_flow w_disp 含义

Boomer1.101.101.001.05近距离支援型，整体权重温和。

权重 CVar 描述已按源码 enum 和解析顺序对齐为 `(S,B,H,P,J,C)`。调权重时按 Smoker、Boomer、Hunter、Spitter、Jockey、Charger 理解。

特感刷新喜好

Smoker 甜点 60%高点优先 偏远、偏高、Flow 稳定。技能未就绪时不会被传送。
Boomer 甜点 25%近点支援 偏近，支援类受 `SUPPORT_EXPAND_MAX=1200` 限制。
Hunter 甜点 45%高度有利 中远距离，杀手/优先解锁类之一。
Spitter 甜点 40%Flow 和分散 支援类，传送时会尊重喷酸后 2 秒间隔。
Jockey 甜点 35%同平面 不吃高度 ring slack，高低差过大扣高度分。
Charger 甜点 38%冲线角度 不吃高度 ring slack，更偏能直接压到人的平面点。
支援门槛：`Smoker/Boomer/Spitter` 会被 `IsSupportGateClassInt` 纳入门槛控制；默认需要 Hunter/Jockey/Charger 这类优先特感达到一定数量或等待 grace 后才放开。

anti-baiter 流程

anti-baiter 不是直接无脑补波，它先判断队伍是不是停滞抱团，并且开新波仍受普通补波 floor 控制。

Off

未开插件、未开始刷特、暂停、Tank、多人倒地或全员被控。

Grace

每波开始后默认 8 秒宽限，建立平均和最高 Flow 基线。

Observe

观察是否 12 秒没有推进，同时队伍仍互相覆盖。

Pressure

停滞且处于压力窗口。可限制最晚开波，并允许不可见 SI 快速传送。

Recover

队伍推进或散开后进入恢复，默认 4 秒后回 Observe。

进入 Pressure 需要
超过 grace + window、团队抱团、没有推进、场上有压力或已到可判定窗口。
不会绕过 floor
`ShouldStartWaveEarly()` 要求 `DifficultyStrategy_CanStartNormalWave()`，所以它只是请求结束等待，不是新增无下限补波。

跑男流程

采样进度每秒取存活未倒地生还者 Flow，算 max、median、avg。
领先判定最高进度比中位数高 `inf_antibait_runner_gap_pct`，默认 12%。
孤立判定1200 范围内没有其他存活未倒地生还者，才算跑男。
动作设置 `rushManIndex`，发 `OnDetectRushman`，目标选择优先跑男，传送可走 1.5 秒快通道。
跑男快通道只影响不可见存活 SI 的传送换点，不会直接增加普通补波数量。

bug 审计摘要

P1：成功刷出后重复计数
`DoSpawnAt()` 已 `RecalcSiCapFromAlive(false)`，成功分支又手动 `siAlive++ / totalSI++`。6 特可能实际刷 3 只就被误判满。
P2：ResetMatchState 未清 siAlive
重复调用会累加旧 `siAlive[]`，影响 cap 和中间状态。
P2：传送队首堵塞
队首类别达到限类时直接 return，可能挡住后续传送项和普通刷特。
P3：文档/调试偏差
低存活阈值、`nb_assault` bypass、anti-bait debug、fallback reset、权重顺序都需要对齐。
